O que faz

Tela mínima que aparece IMEDIATAMENTE quando o paciente clica no link. Mostra logo VITAE + spinner. Por baixo, o JavaScript executa decidirTela() que olha o estado do paciente (já viu slides? logado? perfil completo?) e troca pra tela correta.

✓
Regras
  • Aparece antes do navegador desenhar qualquer outra tela. Body começa com visibility:hidden, só a tela de loading é visível.
  • Tempo máximo visível: 500ms. Se demorar mais, mostra "Carregando sua pré-consulta…" pra dar feedback.
  • Tempo mínimo visível: 150ms. Se decisão for instantânea, ainda esperamos 150ms pra não parecer "piscou".
  • Se decidirTela() falhar (raro), mostra tela de erro com botão "Tentar de novo".
→
Cenários
  • Primeira vez: loading → 1º onboarding
  • Já viu slides, não logado: loading → login gate
  • Logado, perfil incompleto: loading → quiz de perfil (CPF, data)
  • Logado, perfil OK, viu onb2: loading → quiz V4 direto
  • Retomando do meio: loading → quiz V4 na pergunta exata onde parou
  • Já enviou antes: loading → tela "Tudo enviado, pode fechar"
⊘
Bug que isto previne

Hoje (sem essa tela): paciente abre o link e vê a tela de gravar piscar por 1-2 segundos antes do redirect pro 1º onboarding. Causa: navegador desenha a tela inteira antes do JavaScript decidir pra onde mandar.

Com essa tela: o body começa escondido. Só a logo+spinner aparece. Quando decidirTela() termina, o body é revelado já com a tela certa. Zero flicker.

O que faz

Tela idêntica ao 03-cadastro.html que você já tem em produção — mesmos componentes, mesmo estilo, mesmo comportamento. Mas dentro do arquivo único da pré-consulta (não é mais redirect pra outro arquivo).

Tem 2 modos: Cadastro novo (campos completos) e Login (só email+senha, ativado pelo link "Já tem conta? Entrar").

✓
Regras de comportamento
  • Aparece sempre que estado.logado = false (exceto se paciente já enviou pré-consulta antes — aí vai direto pro "Tudo enviado").
  • Botão "Criar conta" só fica ativo quando: nome preenchido + celular válido + email válido + senha ≥ 8 + LGPD marcado.
  • "Continuar com Google" → modo redirect (sai do app pro navegador do Google, volta limpo). Funciona em WhatsApp, iPhone, Android, qualquer browser.
  • "Já tem conta? Entrar" → mesmo formulário troca pra modo login (esconde nome+celular, mostra só email+senha + "Esqueci senha").
  • Após sucesso (qualquer caminho): estado.logado = true → decidirTela() → próximo gate (perfil completo? → 2º onboarding ou quiz-perfil).
⊘
Bug do Google iOS resolvido

Problema atual: botão Google usa popup. No WhatsApp do iPhone, o popup é suspendido pelo iOS quando o foco muda. Callback nunca dispara. Paciente "perde" o login (foi o que aconteceu com seu sócio).

Solução nova: trocar pra modo redirect (mais robusto). Paciente sai do app, faz Google no navegador completo, volta. É 1 segundo mais lento mas nunca falha em nenhum browser, nenhum celular.

→
Cenários de erro tratados
  • Email já cadastrado: mensagem inline "Esse email já tem conta. Quer entrar?" + botão direto pra modo login com email pré-preenchido.
  • Senha curta/fraca: aviso embaixo do campo enquanto digita ("Falta 3 caracteres", "Adiciona um número pra ficar mais segura").
  • Celular inválido: máscara automática (11) 99999-9999. Se ficar incompleto ao perder foco, fica vermelho.
  • LGPD desmarcado: botão fica acinzentado e desabilitado. Tooltip "Aceite os termos pra continuar".
  • Email errado no login: "Email ou senha incorretos" embaixo do botão. Não pula tela, não limpa campos.
  • Cancelou o Google: volta pra essa tela com tudo intacto. Sem mensagem de erro (não foi erro, foi escolha).
  • Google falhou (rede/auth): tela mostra "Não consegui conectar com Google. Tenta de novo ou use email."
  • Voltar do navegador: volta pro 1º onboarding (mas com slides já vistos = pula direto pra cá de novo).
!
Pegadinhas históricas que fica de olho
  • redirect_uri_mismatch: Sessão 14 documentou esse erro 400 do Google. Solução: usar exatamente a mesma URL configurada no Google Cloud Console (já tem https://vitae-app.vercel.app/).
  • Token de retorno perdido: hoje o token vai por localStorage vitae_quiz_retorno. Pode sumir se navegador limpar cache. Solução nova: token fica também na URL ?retorno=TOKEN, redundante.
  • Loop voltando pros slides: hoje uma flag escrita com nome diferente do que é lida. Solução: estado consolidado em uma chave única.

!
O bug que você sentiu hoje (falso "sem internet")

O que aconteceu: você tava com WiFi normal e o app falou que tava sem internet.

Causa raiz encontrada: o V4 atual usa um detector chamado navigator.onLine. Esse detector é notoriamente não-confiável em todos os navegadores — dispara "ficou offline" quando:

  • Celular troca de rede (WiFi → 4G ou vice-versa) por 1 segundo
  • iOS suspende o JavaScript quando bloqueia a tela
  • Você passa por túnel ou área de sinal fraco por 2 segundos
  • WhatsApp em background "congela" o navegador interno
  • VPN reconecta sozinha

Em todos esses casos, sua internet está OK, mas o navegador grita "OFFLINE!" por engano.

⊘
Solução nova (vai eliminar o falso positivo)

Em vez de confiar no detector ruim do navegador, vamos validar de verdade: quando o navegador disser "ficou offline", o sistema faz uma checagem rápida (1 segundo) com nosso servidor antes de mostrar o banner.

  • Servidor responde rápido → era falso positivo, banner não aparece
  • Servidor não responde → realmente sem internet, mostra o banner
  • Banner sempre tem botão "Verificar conexão de novo" pra paciente forçar nova checagem
  • Mesmo "sem internet de verdade", paciente pode continuar respondendo: tudo salva no celular e sincroniza depois

Bugs prevenidos pela arquitetura nova
Cada bug que você reportou + cada gambiarra catalogada. O que sai de cena.
Bug A · Google iOS
Login Google quebrava no WhatsApp
Popup era suspendido pelo iOS quando o foco mudava. Callback nunca disparava.
Trocado pra modo redirect (sai do app, volta limpo). Funciona em qualquer celular.
Bug B · Flicker
Tela de gravar piscava por 1-2s
DOM inteiro era pintado antes do JavaScript decidir o redirect.
Body começa escondido. Loading screen aparece imediato. Decisão acontece por baixo.
Bug C · Pulos aleatórios
Telas trocavam sem motivo
Flag de "slides visto" era escrita com nome diferente do que era lida (chaves desacopladas).
Estado consolidado em 1 objeto único. Mesma chave escrita e lida em todo lugar.
Bug D · Sensação de remendo
Código fragmentado em 4 arquivos
7 redirects encadeados, 12 chaves de localStorage, 13 gambiarras catalogadas.
1 arquivo só. 3 chaves. 0 redirect. Cada bug futuro tem 1 lugar pra olhar.
Bug E · 2 botões redundantes
"Começar gravando" / "Prefiro digitar" duplicados
Faziam exatamente a mesma coisa quando V4 estava ativo. Código zumbi.
1 botão único "Começar pré-consulta →". Sem ambiguidade.
Gambiarra 1 · Função mentirosa
"showMainContent" que redirecionava
Nome dizia uma coisa, código fazia outra. Confunde qualquer dev que mexer depois.
Função única "decidirTela()" que faz exatamente o que o nome diz.
Gambiarra 2 · 3 sistemas de versão
V1, V2 e V4 rodando juntos
8 combinações possíveis de estado. Pra entender qual versão ativa, precisava ler 2700 linhas.
Só V4 existe. V1 e V2 movidos pra pasta legacy/.
Gambiarra 3 · 2 telas de login
Login dentro do arquivo + 03-cadastro
Comportamentos pós-login diferentes. Causa parte dos pulos aleatórios.
Login inline único. 03-cadastro só é usado pro Google (modo isolado).
Gambiarra 4 · Try/catch silenciosos
Erros engolidos sem log
Quando algo falhava, código simulava sucesso. Paciente via tela errada e ninguém sabia o porquê.
Erros logados com contexto. Falha sempre volta pro estado mais conservador (login ou loading).
Gambiarra 5 · Código V1 morto
Painel de gravação V1 ainda renderizava
DOM completo do V1 era pintado mesmo quando ia redirecionar pro V4 dali a 1s. Causa o flicker.
V1 deletado. Só V4 existe no DOM.
Bug F · Falso "sem internet"
App falava que tava offline com internet OK
Detector navigator.onLine do navegador é não-confiável. Dispara false positive ao trocar rede, passar em túnel, iOS suspender JS.
Validação real com servidor antes de mostrar banner. Falso positivo nunca mais.
Pegadinha 1 · Mic em iOS
Permissão de mic perdida ao trocar de tela
iOS Safari revoga permissão do mic se navegador entrar em background mais de 30s. Próxima gravação falha silenciosa.
Sistema reverifica permissão ANTES de cada gravação. Se perdeu, mostra sheet acolhedor pra liberar.
Pegadinha 2 · IndexedDB compartilhado
V2 e V4 usam mesmo banco local
Hoje, dado salvo numa versão pode aparecer mal-formatado na outra (causa pulos aleatórios).
Banco local com versão única + namespace por token. Sem mistura entre versões.
Pegadinha 3 · Bluetooth no meio
AirPod desconectou enquanto gravava
MediaRecorder solta erro silencioso. Resposta perdida sem aviso pro paciente.
Detecta erro do recorder, salva o que tem, retoma com mic do celular automaticamente.
Pegadinha 4 · Ligação chegou
App pausa, paciente atende, volta
iOS suspende JavaScript ao atender ligação. Gravação corrompe.
visibilitychange detecta pausa, salva chunks parciais, oferece "Retomar de onde parou" ao voltar.
