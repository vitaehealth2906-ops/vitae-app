i
O que faz

Tela mínima que aparece IMEDIATAMENTE quando o paciente clica no link. Mostra logo VITAE + spinner. Por baixo, o JavaScript executa decidirTela() que olha o estado do paciente (já viu slides? logado? perfil completo?) e troca pra tela correta.

✓
Regras
  • Aparece antes do navegador desenhar qualquer outra tela. Body começa com visibility:hidden, só a tela de loading é visível.
  • Tempo máximo visível: 500ms. Se demorar mais, mostra "Carregando sua pré-consulta…" pra dar feedback.
  • Tempo mínimo visível: 150ms. Se decisão for instantânea, ainda esperamos 150ms pra não parecer "piscou".
  • Se decidirTela() falhar (raro), mostra tela de erro com botão "Tentar de novo".
→
Cenários
  • Primeira vez: loading → 1º onboarding
  • Já viu slides, não logado: loading → login gate
  • Logado, perfil incompleto: loading → quiz de perfil (CPF, data)
  • Logado, perfil OK, viu onb2: loading → quiz V4 direto
  • Retomando do meio: loading → quiz V4 na pergunta exata onde parou
  • Já enviou antes: loading → tela "Tudo enviado, pode fechar"
⊘
Bug que isto previne

Hoje (sem essa tela): paciente abre o link e vê a tela de gravar piscar por 1-2 segundos antes do redirect pro 1º onboarding. Causa: navegador desenha a tela inteira antes do JavaScript decidir pra onde mandar.

Com essa tela: o body começa escondido. Só a logo+spinner aparece. Quando decidirTela() termina, o body é revelado já com a tela certa. Zero flicker.

O que faz

Login obrigatório. 3 caminhos: (1) Continuar com Google — caminho principal, mais rápido. (2) Email + senha — pra quem já tem vita id. (3) Criar com email — pra quem prefere não usar Google.

Selo de LGPD logo no topo combate a objeção "vou dar meus dados pra quem?".

✓
Regras
  • Aparece sempre que logado: false, exceto se já enviou pré-consulta antes.
  • Email + senha → vitaeAPI.login() → estado.logado = true → decidirTela() → próxima etapa.
  • Google → redireciona breve pra 03-cadastro.html?retorno=TOKEN&googleOnly=1 (modo só-Google) → após autenticar, volta pra pre-consulta.html?token=TOKEN → estado restaurado, decidirTela() → próxima etapa.
  • "Criar com email" → leva pro 03-cadastro.html?retorno=TOKEN normal (cadastro completo) → volta limpo.
  • Em todos os casos, ao retornar do 03-cadastro, o estado vitae_pc_state está intacto. Slides já vistos não aparecem de novo.
⊘
Bug do Google Sign-In resolvido

Problema atual: botão Google usa popup. No WhatsApp do iPhone, o popup é suspendido pelo iOS quando o foco muda. Callback nunca dispara. Paciente "perde" o login.

Solução nova: O botão Google sai do popup. Vai pro 03-cadastro.html?googleOnly=1 que usa o método redirect (paciente sai do app, faz Google no navegador completo, volta). Funciona em iOS, Android, WhatsApp, qualquer ambiente. É 1 segundo mais lento mas nunca falha.

→
Cenários de erro tratados
  • Email errado: mensagem inline "Email ou senha incorretos" abaixo do botão. Não pula tela.
  • Cancelou o Google: volta pra essa tela com tudo intacto. Sem mensagem de erro (não foi erro, foi escolha).
  • Google falhou (rede/auth): tela do 03-cadastro mostra "Tente de novo ou use email". Volta pra cá ao desistir.
  • Já tem conta com esse email mas tentou criar: 03-cadastro detecta e oferece "Você já tem conta. Fazer login?".
